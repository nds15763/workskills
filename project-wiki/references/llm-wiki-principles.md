# LLM Wiki 核心原则

## 核心洞察

> 知识只编译一次并保持更新，而不是在每次查询时重新推导。

这是 project-wiki 和普通搜索工具（RAG/rg/grep）的根本区别：

- **普通搜索**：每次从零检索、拼接、推导 → 没有积累，知识不产生复利
- **LLM Wiki**：LLM 增量构建并维护一个持久化 Wiki → 交叉引用已存在，矛盾已标记，综合分析已反映一切

## 角色分工

**Obsidian 是 IDE，LLM 是程序员，Wiki 是源代码。**

| 角色 | 职责 |
|------|------|
| 人类 | 策展素材、引导分析、提出好问题、确权灰知识 |
| LLM | 摘要、交叉引用、归档、记账——让知识库长期有效的所有繁杂工作 |
| Obsidian | 存储、链接、搜索——知识基础设施 |

## 三层分离原则

### Layer 1: Raw Sources（不可变）

- 存放于 `_sources/`
- 包括：PRD、会议记录、设计文档、论文、文章、代码分析
- **LLM 只读不改**。这是事实之源
- 人类负责策展——决定什么值得进入知识库

### Layer 2: Wiki（LLM 维护）

- 存放于 `<业务域>/` 目录
- 包括：域索引、实体页、概念页、知识沉淀、对比表、综合分析
- **LLM 完全拥有这一层**：创建、更新、重组、维护链接
- 每个业务域 = 一组 wiki 页面
- Wiki 页面是 **编译产物**，不是原始素材的副本

### Layer 3: Schema（治理规则）

- 存放于 `_schema/`（vault 级覆盖）和 skill 的 references/
- 包括：命名规范、页面模板、读写合同、三色沉淀规则
- 人类和 LLM 共同演进这一层

## 知识复利模型

### 为什么知识会产生复利

1. **交叉引用已存在**：新素材进入时，LLM 自动建立与已有知识的链接
2. **矛盾已标记**：冲突在摄取时就被发现并记录，不是在查询时才暴露
3. **综合分析持续演进**：每次摄取都强化或挑战正在演进的论点
4. **查询结果可归档**：一个你要求的对比表、一个分析、一个发现的关联——这些不消失在聊天历史中

### 复利的实践

| 操作 | 复利效果 |
|------|----------|
| Ingest | 一个源素材可能更新 10-15 个 wiki 页面 |
| Query | 好的查询结果归档为新页面，丰富知识网络 |
| Lint | 发现空白和矛盾，引导下一步调研方向 |

## index.md 与 log.md

### index.md — 空间维度导航

- 按类别组织（业务域 / 实体 / 概念 / 来源），不按时间
- LLM 在每次 Ingest 时更新
- 查询时先读 index.md 定位相关页面
- 在中等规模（~100 个素材）下，比复杂的向量数据库更有效

### log.md — 时间维度日志

- 每条日志以 `## [YYYY-MM-DD] 操作 | 标题` 开头
- 记录每次操作的新增、更新、冲突、归档
- append-only，可用 grep 快速解析

## 与 Curator 治理的叠加

LLM Wiki 的三层架构提供骨架，Curator 的治理模型叠加在 Layer 2 之上：

- 每条 durable knowledge 必须同时带 knowledge_kind + knowledge_color + authority_level
- 新知识默认进入 gray（待确权）
- 确权后升级为 white 进入基线
- 弃用后降级为 black 隔离
- 治理规则本身（什么算白、什么算黑）属于 Layer 3

## 不同于 RAG 的地方

| RAG | LLM Wiki |
|-----|----------|
| 每次查询从零检索和拼接 | 知识已编译，查询直接命中 |
| chunk 之间无关联 | wikilink 构成知识网络 |
| 向量相似度驱动 | BM25 + 知识元数据 + authority 排序 |
| 知识只在源文档中 | 知识在 Wiki 层被综合和交叉引用 |
| 查询结果消失在聊天中 | 查询结果可归档为新 wiki 页面 |
| 无矛盾检测 | Lint 主动发现矛盾和空白 |
