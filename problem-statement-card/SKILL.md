> **公理继承 / Axiom Inheritance**
> 本 skill 服从顶层公理 `typed evidence gates action`——
> 未经类型化（三色 + authority + kind）的上下文不允许驱动行动。
> 在该公理下，本 skill 的职能：把开放问题压成带可观测信号、爆炸半径、失效条件的类型化问题陈述（钻石 1 的 Define 收敛）

# Problem Statement Card

## 核心定位：双钻模型的钻石 1 工具

```
┌──── 钻石 1 ────┐  ┌──── 钻石 2 ────┐
Discover   Define     Develop    Deliver
（侦察）   （定义）     （发散）   （落地）
   ↑          ↑           ↑          ↑
找事实    锁问题      列解法    选一个
   └────────┘            └────────┘
   本 skill 的领地      knowledge-card-qa 等的领地
```

**钻石 1 未完成 → 钻石 2 必然内耗**。本 skill 唯一目的：把模糊期望压成一句话 + 可观测信号，让钻石 2 有筛子可用。

## 何时用 / 何时不用

| 触发信号 | 解读 | 用本 skill |
|---|---|---|
| "我有 N 个解法不知道选哪个" | 跳过钻石 1 直接进钻石 2 | ✅ 先回钻石 1 |
| "功能好不好用 / 好不好看 / 有没有趣" | stakeholder 期望没拆 | ✅ |
| "看不到决策后果" | 没写"可观测的成功/失败信号" | ✅ |
| "新方案效果更好但和旧决策冲突" | 把"反馈"误归类成"冲突" | ✅ 重写陈述卡（不取舍） |
| "激进 vs 保守取哪个" | 性格标签当决策维度 | ✅ |
| "AI 给的证据都对但我还是不踏实" | 钻石 1 没收口，AI 在钻石 2 帮你 | ✅ 拉回钻石 1 |
| 解法已清晰要选哪个 | 这是钻石 2 | ❌ → knowledge-card-qa |
| 旧知识冲突要裁决 | 这是知识治理 | ❌ → project-knowledge-curator |
| 排查 bug / 长材料 review | 这是事实重构 | ❌ → problem-review-mapper |

## 问题陈述卡模板

```md
### 问题陈述卡：<开放问题名>

**How might we** _________________________________________？
（必须能写成一句话；写不出 = 钻石 1 未完成，回 Discover）

**Stakeholder + 真实期望**：
- A（如：用户）：希望 ___
- B（如：我自己 / 团队 / 老板）：希望 ___
- C（如：商业指标 / 转化率）：希望 ___

**可观测的成功信号**（钻石 2 用这个判定 lock 解法对不对）：
- 信号 1：____（必须带数字 / 状态 / 时间窗）
- 信号 2：____
- 信号 3：____

**可观测的失败信号**（反向定义，Johnny 三原则的句法）：
- 信号 1：____
- 信号 2：____

**不做 / 排除**：
- ____
- ____

**反馈周期**：__ 天内能看到信号
**爆炸半径**：失败的最坏后果是 ____（可逆 / 半可逆 / 不可逆）
**注意力预算**：值得花的 attention 上限是 ____

**证据**（Discover 阶段收集的事实）：
- __________

**失效条件**（任一发生 → 重写卡）：
- ___
- ___
- 反馈周期到了但信号都看不到 → 信号定义有问题
- 新方案"效果可能更好" → 反馈触发 Repair Loop
```

## How might we 句式 — 钻石 1 收敛的关键

把模糊期望压成单句：

| 模糊表达 | How might we 改写 |
|---|---|
| "这个功能好不好用" | "How might we 让 X 用户在 Y 场景下 < N 秒完成 Z？" |
| "好不好看" | "How might we 让 X 群体首次看到时点击 CTA 的概率 > N%？" |
| "提升转化率" | "How might we 让 step-A → step-B 的转化率从 N1% 提到 N2%？" |
| "有没有趣" | "How might we 让 X 用户在使用 N 分钟后愿意分享给朋友？" |

**写不出 How might we = 钻石 1 未完成**。继续 Discover，不要硬进钻石 2。

## Johnny 三原则在卡片里的位置

| 三原则 | 卡片字段 | 句法 |
|---|---|---|
| **SSOT 准确性** | Stakeholder 期望 + 成功/失败信号 | 必须客观可证、可对齐 |
| **Fast Feedback** | 反馈周期 | 越短越好，超过周期没信号 → 信号定义错 |
| **Attention Budget** | 爆炸半径 + 注意力预算 | 不可逆 = 高 attention，可逆 = 放手做 |

## 失败模式

- ❌ 跳过 stakeholder 直接写"我想..." → Discover 没做
- ❌ 成功信号写"做好"、"用得舒服"、"有意思" → 不可观测，等于没写
- ❌ How might we 写不出来就直接进解法 → 钻石 1 跳过
- ❌ 把"激进/保守"当决策维度 → 性格标签当 PRD
- ❌ 反馈出现就"取舍"两难 → 应触发 Repair Loop 重写陈述卡
- ❌ 把一张卡塞 3 个 How might we → 一卡一问题，多问题拆多卡
- ❌ "事后否认旧决策"当失败 → 这是 fallibilism 正常工作，不是漏洞

## 生命周期

```
Discover → Define（lock 卡） → 进钻石 2 → 反馈出现
                ↑                              │
                └────── 反馈触发 Repair ───────┘
                       （重写卡，不是取舍）
```

陈述卡是 **gray knowledge by default**——它是当下证据下的最佳问题归纳，未来反馈会让它升 white、降 black 或被替代。任何"事后否认"都是 Repair Loop 在工作，不是失败。

## 落卡判断

继承 `knowledge-card-qa` 的判断标准：

| 场景 | 落不落卡 |
|---|---|
| 一次性、私人决策（"晚饭吃啥"） | 不落 |
| 高频重复（"每次做新功能都纠结好不好看"） | 落 |
| 跨项目复用（"任何转化率优化都套这卡"） | 落到 `~/docs/<域>/problem_statements/` |
| 重大不可逆（创业方向、人生选择） | 必落 + 跨年保留 |

## 公理继承的具体落点

`typed evidence gates action` 在本 skill 表现为：

- **"我有一个想法"** 不能直接驱动行动——必须先类型化成问题陈述卡
- **"AI 说这个对"** 不能直接驱动行动——AI 输出要先过 Discover（喂 Knowledge Pack），再进 Define（写信号）
- **"我直觉觉得"** 不能直接驱动行动——直觉只能进 Discover 当 evidence，不能进 Define 当结论
- **未经写"失败信号"的问题不允许进钻石 2**——这就是 gate

## 元层洞察

如果你发现：**对项目能做完这张卡，对自己人生做不完**——那不是方法论问题，是你**不允许自己的问题陈述被未来重写**。强制承认：**问题陈述卡和解法卡一样可错**。皮尔士的 fallibilism 适用于解法，也适用于问题本身。
